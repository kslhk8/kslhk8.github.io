## 들어가며

하이브리드 앱을 만들다 보면 한 번쯤 이런 생각을 한다.

> "WebView 하나 띄우면 끝 아닌가?"

나도 처음에는 그렇게 생각했다.  
하지만 실제로 서비스를 붙여보면 웹과 앱 사이에서 상태와 기능이 자연스럽게 맞물리지 않는 지점들이 금방 드러난다.

- **앱에만 있는 기능**
  - 푸시 토큰 발급
  - OS 설정 화면 이동
  - 네이티브 공유
  - 권한 확인
- **웹에만 있는 것**
  - Next.js 기반 UI
  - 기존 API와 비즈니스 로직
  - 검색, 마이페이지, 온보딩 플로우
- **둘 다 알아야 하는 상태**
  - 액세스 토큰
  - 최근 검색어
  - 푸시 토큰

이 상태들은 화면이 하나처럼 보여도 자동으로 공유되거나 동기화되지 않는다.  
토큰 전달을 넘어 권한 요청, 공유 기능, 푸시 토큰 처리까지 얽히자 구조가 금방 복잡해졌다.

결국 필요했던 것은 단순한 `WebView 렌더링`이 아니라, **웹과 앱이 상태와 이벤트를 안전하게 주고받는 브릿지 구조**였다.

이 글에서는 브릿지를 역할 중심으로 어떻게 설계했는지 정리해보려 한다.


## 먼저 결론: 내가 선택한 구조

문제를 해결하기 위해 선택한 구조는 **공통 메시지 규약 + 단방향 송수신 경로 + 명시적인 동기화 시점**으로 요약할 수 있다.

이 구조의 핵심은 아래 네 가지다.

- 메시지 타입과 유틸은 공통 브릿지 모듈에서 관리한다 (웹/앱 공통 메시지 형식 유지)
- 웹 → 앱은 `postMessage`로, 앱 → 웹은 `injectJavaScript`로 통신한다 (단방향 경로 고정)
- 브릿지 책임은 웹은 Provider, 앱은 Hook에 모아 관리한다 (로직 분산 방지)
- `WEB_INIT`으로 초기 연결을 시작하고, `TOKEN_REFRESHED`로 런타임 동기화를 유지한다 (상태 동기화 흐름 명시)

이 네 가지 선택을 구조로 그리면 아래와 같다.

```text
Web (Provider)
  ↕
Bridge Layer
  ↕
App (Hook)
  ↕
SecureStore / Native APIs / Device State
```

이 구조는 웹과 앱의 책임이 어디서 나뉘는지를 먼저 보여준다.

그리고 이 구조 안에서 실제로 오가는 메시지는 역할 기준으로 세 묶음으로 나눴다. 이렇게 구분해두니 초기화 메시지, 상태 동기화 메시지, 네이티브 기능 호출 메시지가 서로 섞이지 않았다.

### 메시지 분류

- `Lifecycle`
  - `WEB_INIT`: 웹이 초기 연결 시작을 알린다
- `Sync`
  - `ACCESS_TOKEN`: 앱이 초기 인증 정보를 전달한다
  - `RECENT_SEARCHES`: 앱이 초기 최근 검색어를 전달한다
  - `TOKEN_REFRESHED`: 웹이 런타임 토큰 갱신을 알린다
  - `PUSH_ALARM`: 앱과 웹이 푸시 동의 상태와 토큰을 맞춘다
- `Action`
  - `CHECK_MEDIA_PERMISSION` / `MEDIA_PERMISSION_RESULT`
  - `OPEN_APP_SETTINGS`
  - `SHARE`
  - `SIGN_OUT`
  - `SET_RECENT_SEARCHES`

이제부터는 왜 이 구조가 필요했고, 실제로 어떤 흐름으로 동작하는지 순서대로 보겠다.

## 왜 WebView만으로는 부족했는가

겉으로는 간단해 보였지만, 실제로는 네 가지 문제가 드러났다.

### 1. 같은 화면처럼 보여도 상태가 자동으로 공유되지는 않는다

WebView 안에 웹을 띄우면 겉으로는 하나의 화면처럼 보이지만, 실제로는 웹과 앱이 서로 다른 환경에서 동작한다.

핵심은 UI만 함께 보일 뿐, React 상태와 저장소를 포함한 실행 환경은 서로 분리되어 있다는 점이다.

### 2. 토큰은 한 번 전달로 끝나지 않는다

초기 진입 시점의 토큰만 넘기면 끝날 것 같지만, 실제 서비스에서는 **토큰 갱신**이 발생한다.

웹 쪽에서 토큰이 갱신됐는데 앱이 예전 토큰을 계속 들고 있으면,

- 앱 재실행 시 오래된 토큰을 웹으로 다시 주입할 수 있고
- 앱 기준 인증 상태와 웹 기준 인증 상태가 어긋날 수 있다

실제로 이 문제 때문에 로그인 상태가 어긋나는 상황도 있었다.

### 3. 브릿지는 순서가 어긋나면 쉽게 꼬인다

웹이 메시지를 보냈다고 해서 앱이 항상 받을 준비가 된 것은 아니다.  
반대로 앱이 웹으로 메시지를 주입한다고 해서 웹에 핸들러가 이미 등록되어 있다고 보장할 수도 없다.

브릿지에서는 기능 자체보다 **누가 먼저 준비되는지와 어떤 순서로 주고받는지**가 더 중요할 때가 많다.

### 4. 기능이 늘어날수록 분산 구현이 위험해진다

처음에는 `공유하기 하나`, `권한 요청 하나` 정도로 보이지만, 기능이 늘어나면 인증, 권한, 네이티브 기능, 상태 동기화 메시지가 한 흐름 안에서 얽힌다.

이걸 각 화면에서 제각각 구현하면 **어디서 어떤 메시지를 보내고 받는지 추적하기 어려워진다.**

## 시도했지만 선택하지 않은 방법

### 1. URL 파라미터나 초기 헤더로 모든 값을 넘기기

예를 들어 `inApp=true` 같은 헤더나 쿼리스트링을 붙여서 웹이 앱 내부라는 사실을 알게 만드는 방식이다.

실제로 WebView를 띄울 때도 이런 식의 값은 함께 실어줄 수 있다. 이런 정보는 환경 식별 용도로는 유용하다.

하지만 이것만으로는 부족했다.

- 토큰 갱신을 반영하기 어렵다
- 푸시 토큰처럼 앱에서 나중에 비동기로 얻는 값은 전달하기 어렵다
- 권한 상태처럼 사용자 행동에 따라 달라지는 값을 실시간으로 주고받기 어렵다

즉, **초기 진입 정보를 알려주는 데는** 쓸 수 있어도, **이후 상태를 계속 맞춰 주는 방식으로는** 부족했다.

### 2. 웹과 앱이 각자 알아서 저장하고 맞춰보는 방식

이 접근은 결국 동기화 문제를 만든다.

- 앱은 `SecureStore`
- 웹은 `localStorage` 또는 상태 스토어

저장소가 분리되어 있기 때문에, 둘 중 하나가 갱신되면 다른 한쪽에 **명시적으로 알려주는 단계**가 반드시 필요하다.

### 3. 화면마다 `postMessage`를 직접 호출하는 방식

초반에는 가장 빨라 보인다. 하지만 금방 문제가 생긴다.

- 메시지 포맷이 파일마다 달라진다
- 메시지 이름이 문자열로 흩어진다
- 어디서 수신 중인지 추적하기 어렵다
- 디버깅할 때 진입점을 찾기 힘들다

결국 더 중요했던 건 `postMessage`를 쓰는 것 자체보다, **메시지 형식과 송수신 흐름을 한 곳에서 관리하는 구조**였다.

## 최종 구조를 나눠서 보면

앞에서 요약한 구조를 조금 더 구체적으로 풀면 다음과 같다.

### 1. 웹과 앱이 함께 쓰는 공통 브릿지 모듈을 둔다

여기서 말하는 공통 브릿지 모듈은, 웹과 앱이 함께 참고하는 메시지 타입과 브릿지 유틸을 모아둔 레이어를 뜻한다.

예를 들어 `WebViewMessage` 같은 메시지 타입과, 웹에서 앱으로 메시지를 보내는 함수가 여기에 들어간다.

이 선택이 중요했던 이유는, 웹과 앱이 서로 다른 메시지 형식을 기준으로 움직이기 시작하는 순간부터 문제가 커지기 때문이다.

공통 브릿지 모듈에 메시지 형식과 규칙을 모아두면 다음이 가능해진다.

- 웹과 앱이 같은 메시지 구조를 바라본다
- 메시지 확장 시 영향 범위를 파악하기 쉽다
- 브릿지 유틸을 한 곳에서 재사용할 수 있다

### 2. 웹 -> 앱은 `window.ReactNativeWebView.postMessage`

웹에서 앱으로 메시지를 보낼 때는 `sendMessageToNative` 같은 공통 함수를 두고, 그 안에서 아래 흐름을 감싸는 방식으로 정리했다.

- 브라우저 환경인지 확인한다
- `window.ReactNativeWebView` 존재 여부를 확인한다
- 메시지를 `JSON.stringify` 해서 `postMessage`로 보낸다

코드로 보면 흐름은 아주 단순하다.

```ts
export const sendMessageToNative = (message: WebViewMessage): void => {
  if (typeof window === "undefined") return

  if (window.ReactNativeWebView) {
    window.ReactNativeWebView.postMessage(JSON.stringify(message))
  }
}
```

여기서 더 중요한 건 `JSON.stringify` 자체보다, 이 직렬화 규칙을 공통 유틸 안에서 일관되게 처리하도록 만든 점이다.

그래야 메시지 형식이 흔들리지 않고, 메시지를 보내는 쪽 코드도 단순하게 유지할 수 있다.

### 3. 앱 -> 웹은 `injectJavaScript`로 전역 핸들러를 호출한다

React Native WebView는 웹이 앱으로 보내는 `postMessage`는 잘 지원하지만, 반대 방향은 `window.postMessage`처럼 대칭적이지 않다.

그래서 앱 쪽 `useWebViewBridge`에서는 다음 방식을 사용한다.

- `WebView` ref를 잡아둔다
- `injectJavaScript`로 웹 컨텍스트에 코드를 주입한다
- 주입된 코드가 `window.handleNativeMessage(message)`를 호출한다

앱 쪽에서는 이 부분이 실제 전송 함수 역할을 한다.

```ts
const sendToWeb = useCallback((message: WebViewMessage) => {
  if (!webViewRef.current) return

  const script = `
    if (window.handleNativeMessage) {
      window.handleNativeMessage(${JSON.stringify(message)});
    }
  `

  webViewRef.current.injectJavaScript(script)
}, [])
```

웹 쪽에서는 `window.handleNativeMessage`를 등록하고, 내부적으로 구독 중인 콜백들을 순회하면서 메시지를 전달한다.

여기서 중요한 건 `injectJavaScript`라는 도구 자체보다, 앱이 웹으로 값을 전달하는 방식을 일관되게 처리했다는 점이다.

덕분에 앱에서 어떤 메시지가 웹으로 들어가는지 추적하기 쉬워졌고, 웹 쪽에서는 이를 구독 형태로 단순하게 다룰 수 있었다.

### 4. 웹은 Provider에서, 앱은 Hook에서 책임을 모은다

`WebViewBridgeProvider`는 웹 쪽에서 브릿지 처리를 맡는 중심 역할이다.

이 Provider가 맡는 일은 크게 세 가지다.

- 현재 환경이 WebView인지 판별한다
- 초기 연결을 시작한다
- 앱에서 온 메시지를 받아 웹 상태에 반영한다

즉, 웹에서는 브릿지의 시작과 수신 처리를 Provider가 맡는다.

반대로 `useWebViewBridge`는 앱 쪽에서 브릿지 처리를 맡는 Hook이다.

이 Hook은 다음 역할을 맡는다.

- 웹에서 온 메시지를 파싱한다
- 네이티브 기능으로 연결한다
- 필요한 값을 다시 웹으로 전달한다

즉, 앱에서는 네이티브 기능 실행과 응답 전송을 Hook이 맡는다.

## 초기 연결과 상태 동기화

이 구조에서 가장 중요했던 것은 `초기 연결 과정`의 기준점을 명확하게 잡는 일이었다.

### 1. 웹이 WebView 환경인지 확인한다

`WebViewBridgeProvider`는 마운트 시 `isInWebView()`를 호출한다.

여기서 중요한 점은 **웹 코드가 브라우저와 앱 WebView 양쪽에서 실행될 수 있다**는 사실이다. 따라서 모든 브릿지 로직은 환경 감지가 먼저여야 한다.

### 2. 웹이 앱에 `WEB_INIT` 메시지를 보낸다

웹이 먼저 `WEB_INIT`을 보내는 이유는, 앱이 임의 시점에 값을 주입하는 방식보다  
**웹이 받을 준비가 되었다는 신호를 먼저 보내는 편이 안전하기 때문**이다.

실제 코드는 아래처럼 매우 짧다.

```ts
if (inWebView) {
  sendMessageToNative({ type: "WEB_INIT" })
}
```

이 단계가 없으면 흔히 아래 문제가 생긴다.

- 앱은 이미 메시지를 보냈는데 웹 핸들러가 아직 없다
- 웹은 토큰을 기다리는데 앱은 언제 보내야 하는지 모른다

즉, `WEB_INIT`은 단순한 시작 메시지가 아니라 **초기 동기화의 기준점**이다.

### 3. 앱이 초기 상태를 웹으로 보낸다

`useWebViewBridge`에서 `WEB_INIT`을 받으면 `handleWebInit()`이 실행된다.

여기서 앱은 웹에 필요한 초기 상태를 모아서 보낸다.

- `ACCESS_TOKEN`
- `RECENT_SEARCHES`
- `PUSH_ALARM`(초기 권한 확인 후 푸시 토큰이 있으면)

앱 저장소인 `SecureStore`와 푸시 권한 상태를 읽어와,  
웹이 처음 화면을 준비하는 데 필요한 정보만 전달하는 구조다.

### 4. 웹이 상태를 반영하고 후속 작업을 수행한다

웹의 `WebViewBridgeProvider`는 `onNativeMessage`로 메시지를 구독한다.

수신한 메시지에 따라 다음 작업을 수행한다.

- `ACCESS_TOKEN` 수신
  - 웹 상태 스토어에 토큰 저장
  - 사용자 정보 조회 가능 상태로 전환
- `RECENT_SEARCHES` 수신
  - 웹 `localStorage`에 최근 검색어 저장
- `PUSH_ALARM` 수신
  - 푸시 토큰을 서버에 등록
  - 필요 시 푸시 수신 동의 상태도 함께 업데이트

이 흐름 덕분에 웹은 **앱이 주는 초기 상태**를 기준으로 화면을 초기화할 수 있다.

## 런타임 동기화

앞서 말했듯 초기 진입 시점 동기화만으로는 충분하지 않았다. 런타임에도 최신 상태를 계속 맞춰야 했다.

초기에는 `ACCESS_TOKEN`만 전달하면 충분해 보인다. 하지만 웹은 API 호출 중 `401`이 나면 자체적으로 토큰을 갱신한다.

웹의 axios 인터셉터에서는, 토큰 갱신에 성공하면 웹 스토어의 토큰을 바꾼 다음 WebView 환경이면 앱으로도 아래 메시지를 보낸다.

- `TOKEN_REFRESHED`

```ts
if (inWebView) {
  sendMessageToNative({
    type: "TOKEN_REFRESHED",
    data: newAccessToken,
  })
  return Promise.reject(error)
}
```

이 메시지를 앱의 `useWebViewBridge`가 받아서,

- 앱 상태 스토어를 갱신하고
- 토큰 캐시에 다시 저장한다

이 단계가 없으면 **웹의 최신 토큰과 앱이 들고 있는 토큰이 다시 어긋난다.**

그래서 인증은 값을 한 번 주입하는 작업이 아니라, `WEB_INIT`으로 시작하고 `TOKEN_REFRESHED`로 이어지는 동기화 흐름으로 다뤄야 했다.

## 브릿지가 실제로 맡은 일

이번 브릿지는 인증 토큰 하나만 주고받는 데서 끝나지 않았다. 웹에서 필요할 때 네이티브 기능을 호출할 수 있게 해주는 메시지 구조로도 동작했다.

### 1. 권한 처리

웹에서 이미지 선택이 필요하면 `CHECK_MEDIA_PERMISSION`을 앱으로 보낸다.

앱은 Expo Image Picker 권한 상태를 확인한 뒤 `MEDIA_PERMISSION_RESULT`를 다시 웹으로 전달한다.

웹은 브라우저처럼 동작하면서도, 실제 권한 확인은 네이티브 API를 통해 처리하도록 구성했다.

권한이 영구 거부된 경우에는 `OPEN_APP_SETTINGS`를 통해 설정 화면으로 이동시키는 흐름도 구성했다.

### 2. 네이티브 공유

웹은 공유하기 버튼 클릭 시 `SHARE` 메시지를 앱으로 전달한다.

실제 공유 UI는 앱이 `Share.share()`를 통해 실행한다.

웹은 사용자 인터랙션을 담당하고, 앱은 OS 기능을 수행하도록 역할을 분리했다.


### 3. 푸시 토큰 전달과 서버 반영

웹은 특정 시점에 `PUSH_ALARM` 메시지를 앱으로 보내고, 앱은 권한 요청 후 푸시 토큰을 획득한다.

이후 앱이 동일한 `PUSH_ALARM` 메시지로 토큰을 웹에 전달하면,  
웹은 서버 API를 호출해 디바이스 토큰과 푸시 수신 동의 상태를 업데이트한다.

디바이스 의존 로직은 앱에서 처리하고,  
서버와 연동되는 비즈니스 로직은 웹에서 처리하도록 역할을 분리했다.

### 4. 로그아웃

웹에서 로그아웃이 발생하면 `SIGN_OUT` 메시지를 앱으로 전달해 앱의 인증 상태도 함께 초기화한다.

웹과 앱 간 인증 상태를 동일한 흐름으로 관리할 수 있도록 구성했다.

## 구현 원칙

결과적으로 이 구조에서 중요했던 기준은 두 가지였다.

### 1. 인증은 값 전달이 아니라 동기화 흐름이다

`ACCESS_TOKEN`은 초기 진입 시 인증 상태를 맞추는 역할을 하고,  
`TOKEN_REFRESHED`는 런타임에 바뀐 토큰을 다시 동기화하는 역할을 한다.  
둘은 기능이 같지는 않지만, 함께 있어야 앱 재실행이나 토큰 갱신 이후에도 인증 상태가 어긋나지 않는다.

### 2. 브릿지는 화면이 아니라 공통 계층에서 다룬다

권한 요청, 공유, 토큰 처리 같은 흐름을 화면마다 흩어두면  
변경 지점과 실패 지점을 함께 추적하기 어려워진다.

이번 구조처럼 웹은 Provider가, 앱은 Hook이 브릿지 처리를 맡도록 두니  
기능이 늘어나도 진입점과 책임이 흔들리지 않았다.

## 마무리

WebView는 화면을 보여주는 레이어였고,  
실제로 필요했던 것은 앱과 웹이 상태와 기능을 같은 흐름 안에서 다룰 수 있게 만드는 브릿지 구조였다.

핵심은 메시지를 한 번 보내는 기술 자체가 아니라,  
어떤 형식으로 주고받을지, 누가 먼저 시작할지, 상태가 바뀌면 언제 다시 맞추지,  
그리고 어느 쪽이 무엇을 책임질지를 미리 정해두는 일이었다.

그렇게 기준점을 먼저 세워두니 웹과 앱은 서로 다른 실행 환경 위에서도 하나의 흐름처럼 동작할 수 있었고,  
기능이 늘어나도 기존 규칙 안에서 안정적으로 확장할 수 있었다.
