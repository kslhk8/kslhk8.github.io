## 들어가며

`PagedGridItemSlider`는 단순한 리스트 UI로는 해결하기 어려운 요구사항에서 시작됐다.

한 화면 안에 여러 개의 아이템을 그리드로 보여주면서,
페이지 단위로 넘기고, 다음 페이지가 살짝 보이는 형태가 필요했다.

여기에 모바일 터치와 데스크톱 드래그까지 자연스럽게 동작해야 해서, 생각보다 고려해야 할 요소가 많았다.

이 글에서는 구현을 세세하게 나열하기보다는
**어떤 문제를 마주했고, 어떤 기준으로 설계했는지**를 중심으로 정리해보려고 한다.

---

## 문제 정의

요구사항은 다음과 같다.

- 한 페이지에 10개 아이템
- 5열 x 2행 그리드
- 좌우 페이지 이동
- 다음 페이지가 살짝 보이는 `peek` UI
- 터치 + 마우스 드래그 지원
- 드래그 중 클릭 오동작 방지

여기서 중요한 점은 이 구조가 일반적인 "아이템 슬라이더"가 아니라는 것이다.

아이템 단위가 아니라 **페이지 단위로 이동하는 UI**였다.

---

## 설계의 기준: 아이템이 아니라 페이지

초기 기준은 명확했다.

**"이 슬라이더는 item이 아니라 page 단위로 동작한다"**

```ts
const PAGE_COLUMN_COUNT = 5
const PAGE_ROW_COUNT = 2
const ITEMS_PER_PAGE = PAGE_COLUMN_COUNT * PAGE_ROW_COUNT
```

```ts
type Item = {
  id: string
  name: string
  logoUrl?: string
}

const chunkItems = (list: Item[]) => {
  const pages: Item[][] = []

  for (let index = 0; index < list.length; index += ITEMS_PER_PAGE) {
    pages.push(list.slice(index, index + ITEMS_PER_PAGE))
  }

  return pages
}
```

이렇게 나누면 이후 로직이 훨씬 단순해진다.

- 현재 위치: 몇 번째 페이지
- 이동 판단: 이전 / 다음 페이지
- 인디케이터: 페이지 개수 기준

UI는 그리드지만, 인터랙션의 기준은 페이지다.

---

## 마지막 페이지 레이아웃 유지하기

페이지형 그리드에서 가장 쉽게 어긋나는 부분은 마지막 페이지다.

예를 들어 17개 데이터라면 마지막 페이지는 7개만 렌더링된다.  
이 상태로 두면 그리드 정렬이 깨진다.

그래서 모든 페이지가 동일한 슬롯 구조를 가지도록 처리했다.

```ts
const createPageSlots = (page: Item[]) => {
  return Array.from({ length: ITEMS_PER_PAGE }, (_, index) => page[index] ?? null)
}
```

이렇게 슬롯 개수를 고정해두면 마지막 페이지도 항상 같은 그리드 구조를 유지할 수 있다.

작은 디테일이지만 레이아웃 안정성을 유지하는 데 중요한 역할을 한다.
---

## peek UI와 이동 거리 계산

이 슬라이더는 다음 페이지가 일부 보이는 구조다.  
그래서 페이지 이동 거리도 단순히 `100%` 기준이 아니라, 실제로 화면에 보이는 페이지 너비를 기준으로 계산해야 한다.

```ts
const LEFT_PADDING = 16

const getPageStep = (container: HTMLDivElement | null, pagePeekWidth: number) => {
  if (!container) return 0

  return Math.max(container.clientWidth - pagePeekWidth - LEFT_PADDING, 0)
}
```

핵심은 **"한 페이지가 보여지는 너비와 이동 거리를 동일한 기준으로 맞추는 것"**이다.

실제 DOM에서는 각 페이지 컨테이너의 `width`를 `calc(100% - pagePeekWidth)`로 설정해
다음 페이지가 일부 보이도록 만들었다.

`getPageStep`에서도 동일한 기준을 사용해 실제 화면과 이동 계산이 어긋나지 않도록 했다.

추가로 paddingRight를 함께 조정해 페이지 간 간격과 정렬이 깨지지 않도록 맞췄다.
---

## transform + ref로 렌더링 비용 줄이기

이동은 CSS `transform`으로 처리했다.

```ts
track.style.transform = `translate3d(${translateX}px, 0, 0)`
```

그리고 드래그 중에 바뀌는 값은 전부 `ref`로 관리했다.

이 조합의 장점은 명확하다.

- 드래그 중에 re-render가 일어나지 않는다.
- GPU 가속으로 이동이 부드럽다.
- React state를 최소한으로만 사용한다.

---

## 클릭 vs 드래그 분리

슬라이더에서 UX가 자주 깨지는 부분 중 하나가 클릭과 드래그가 겹치는 상황이다.  
사용자는 넘기려고 했는데 클릭으로 처리되는 경우가 생긴다.

이걸 막기 위해 **드래그 거리 기준**을 뒀다.

```ts
const DRAG_THRESHOLD = 6

if (dragDistance.current > DRAG_THRESHOLD) {
  return
}
```

핵심은 단순하다.

**"조금 움직인 건 클릭으로 보고,  
충분히 움직였을 때만 드래그로 본다"**

이 기준이 없으면 특히 모바일에서 의도하지 않은 동작이 발생하기 쉽다.

---

## 페이지 전환은 '비율'로 판단

절대 픽셀 기준은 화면마다 체감이 달라진다.  
그래서 페이지 너비 대비 비율로 판단했다.

```ts
const PAGE_CHANGE_THRESHOLD_RATIO = 0.15
```

```ts
const threshold = pageStep * PAGE_CHANGE_THRESHOLD_RATIO
```

이렇게 해두면 큰 화면과 작은 화면 모두에서 비슷한 체감을 유지할 수 있고,
과도하게 민감하거나 둔하지 않은 선에서 페이지 전환을 제어할 수 있다.

---

## 모바일: 가로 드래그인지 먼저 판단

터치 환경에서는 슬라이더보다 중요한 게 하나 있다.

**"세로 스크롤을 방해하지 않는 것"**

그래서 바로 드래그로 취급하지 않고, 방향을 먼저 판단한다.

```ts
if (Math.abs(deltaX) <= Math.abs(deltaY)) {
  isDragging.current = false
  return
}
```

CSS에서도 세로 스크롤을 우선하도록 설정했다.

```tsx
style={{ touchAction: "pan-y" }}
```

이렇게 방향을 먼저 구분하고 `touchAction`까지 함께 설정해두면,
세로 스크롤과 가로 드래그가 서로 덜 충돌하게 된다.
---

## 경계값은 계산으로 처리

슬라이더는 양 끝에서 더 이동하면 안 된다.

```ts
const clampTranslateX = (translateX: number, maxTranslateX: number) => {
  return Math.min(0, Math.max(translateX, -maxTranslateX))
}
```

이렇게 경계값을 제한해두면,
드래그 중에도 이동 값이 양 끝을 넘지 않는다.

---

## state vs ref 역할 분리

상태는 다음과 같이 나눴다.

- `state`: UI에 직접 영향을 주는 값 (`activePage`)
- `ref`: 드래그 중 프레임 단위로 변하지만 렌더링이 필요 없는 값

이렇게 구분하면 드래그처럼 값이 자주 바뀌는 상황에서도 불필요한 렌더링을 줄이면서 UI 상태를 안정적으로 관리할 수 있다.

슬라이더처럼 프레임 단위로 값이 지속적으로 변하는 컴포넌트에서는 이 구분이 중요하다.

---

## Resize 대응

페이지 위치는 컨테이너 크기에 의존하기 때문에 리사이즈 시 현재 위치를 다시 계산해 맞춰줘야 한다.

```ts
new ResizeObserver(() => {
  snapToPage(activePageRef.current, false)
})
```

이 처리를 하지 않으면 화면 회전이나 브라우저 리사이즈 상황에서 페이지 위치가 미세하게 어긋날 수 있다.

---

## 인디케이터는 '조작 UI'로 사용

인디케이터는 현재 페이지를 보여주는 역할과 함께, 각 세그먼트를 클릭해 원하는 페이지로 이동할 수 있도록 구성했다.
```tsx
onClick={() => snapToPage(pageIndex, true)}
```

이렇게 구성하면 사용자가 원하는 페이지로 바로 이동할 수 있다.
---

## 정리

이 컴포넌트에서 핵심이었던 선택들을 정리하면 다음과 같다.

- 페이지 단위로 모델링한다.
- 마지막 페이지도 슬롯 개수를 고정해 레이아웃을 유지한다.
- `peek` UI를 기준으로 실제 이동 거리를 계산한다.
- 클릭과 드래그를 거리 기준으로 분리한다.
- 페이지 전환 기준은 절대값이 아니라 비율로 둔다.
- 모바일에서는 먼저 세로/가로 방향을 구분한다.
- 드래그 중 값은 `ref`로 관리해서 렌더링을 최소화한다.
- 리사이즈 시 현재 페이지 기준으로 위치를 다시 맞춘다.

이런 기준들을 먼저 정리해두니,
레이아웃과 인터랙션도 비교적 일관된 방식으로 맞춰갈 수 있었다.

---

## 마무리

그리드 기반 페이지 슬라이더는 단순히 화면만 움직이는 UI가 아니라,
레이아웃과 드래그 인터랙션, 모바일 터치 대응까지 함께 고려해야 하는 구조였다.
